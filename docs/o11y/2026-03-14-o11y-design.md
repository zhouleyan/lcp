# LCP 可观测性设计

## 概述

LCP 平台的可观测性（O11y）模块，采用 VictoriaMetrics 全家桶方案：

- **Metrics**：node_exporter 采集 → VM all-in-one 抓取并存储 → LCP 代理查询
- **Logs**：vlagent 采集 → VictoriaLogs 存储 → LCP 代理查询
- **Traces**：未来扩展，需引入 OTel Collector 或第三方方案

每台被监控主机部署两个进程：`node_exporter`（指标）+ `vlagent`（日志）。

## 架构总览

```
每台被监控主机:
  ┌───────────────────────────────────────────┐
  │  node_exporter (:9100)   ← 暴露主机指标    │
  │  vlagent                 ← 采集本机日志     │
  │    └── tail /var/log/*   → push 到 VictoriaLogs │
  └───────────────────────────────────────────┘
         ↑ scrape                    │ push
         │                          │
         │                          ↓
  ┌──────┴──────────────────────────────────┐
  │  VM all-in-one                          │
  │  ├── scrape (内置 vmagent) → 存储指标    │
  │  └── query API (/api/v1/query_range)    │
  │                                         │
  │  VictoriaLogs                           │
  │  ├── 接收日志 (HTTP JSON ingestion)     │
  │  └── query API (LogsQL)                 │
  └─────────────────────────────────────────┘
         ↑ http_sd        ↑ proxy query
         │                │
  ┌──────┴────────────────┴─────────────────┐
  │  LCP Server                             │
  │  ├── HTTP SD 接口 (告诉 VM 抓哪些主机)   │
  │  ├── Metrics 代理查询 (PromQL → VM)     │
  │  ├── Logs 代理查询 (LogsQL → VictoriaLogs) │
  │  ├── SSH 安装 node_exporter + vlagent   │
  │  └── 端点管理 + 分配                     │
  └─────────────────────────────────────────┘
```

## 核心决策

| 项 | 决定 |
|---|---|
| Metrics 后端 | VictoriaMetrics all-in-one，管理员在 UI 配置连接地址 |
| Logs 后端 | VictoriaLogs，和 VM 一起部署，管理员在 UI 配置连接地址 |
| Metrics 采集 | node_exporter（行业标准，指标最全） |
| Logs 采集 | vlagent（VM 官方推荐，磁盘缓冲、多目标复制） |
| Metrics 抓取方式 | VM all-in-one 内置 scrape（pull 模式），通过 HTTP SD 获取 target |
| Logs 推送方式 | vlagent 主动推送到 VictoriaLogs（push 模式） |
| 安装方式 | LCP 通过 SSH 远程安装 node_exporter + vlagent |
| SSH 凭证 | 直接加在 Host 表上（端口、用户名、密码/密钥） |
| 安装时机 | 创建主机时可选填 endpointId，也可后续通过 action 按钮接入 |
| 指标展示 | LCP 后端代理查询，前端 recharts 画图，不依赖 Grafana |
| 日志展示 | LCP 后端代理查询 VictoriaLogs，前端日志列表/搜索 |
| Traces | 未来扩展，VM 无 tracing 产品，需引入第三方（OTel Collector + Tempo/Jaeger） |

### 为什么不用 OTel Collector 统一

讨论过用 OTel Collector 单进程替代 node_exporter + vlagent，最终选择 VM 全家桶方案：

| 对比项 | OTel Collector | node_exporter + vlagent |
|--------|---------------|------------------------|
| 主机进程数 | 1 | 2 |
| Metrics 指标丰富度 | hostmetrics receiver 覆盖有限 | node_exporter 几十个 collector、数百个指标 |
| 社区 PromQL 模板 | 很少 | 海量现成 |
| Logs 采集 | filelog receiver 够用 | vlagent 针对 VictoriaLogs 优化 |
| Traces | 原生支持 | 不支持，需第三方 |

当前不需要 traces，metrics 丰富度优先，选 VM 全家桶。未来加 traces 时再评估引入 OTel Collector。

## 数据模型

### o11y_endpoints 表

```sql
CREATE TABLE o11y_endpoints (
    id          BIGSERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL UNIQUE,
    description VARCHAR(500),
    public      BOOLEAN     NOT NULL DEFAULT false,  -- true=全局可用
    metrics_url VARCHAR(500),    -- VictoriaMetrics 查询地址
    logs_url    VARCHAR(500),    -- VictoriaLogs 查询地址
    traces_url  VARCHAR(500),    -- 预留
    status      VARCHAR(20)  NOT NULL DEFAULT 'active',
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

一行 = 一套可观测性栈配置。`metrics_url` 和 `logs_url` 按需填写，未启用的留空。

### o11y_endpoint_assignments 表

```sql
CREATE TABLE o11y_endpoint_assignments (
    id            BIGSERIAL PRIMARY KEY,
    endpoint_id   BIGINT NOT NULL REFERENCES o11y_endpoints(id) ON DELETE CASCADE,
    workspace_id  BIGINT REFERENCES workspaces(id) ON DELETE CASCADE,
    namespace_id  BIGINT REFERENCES namespaces(id) ON DELETE CASCADE,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE(endpoint_id, COALESCE(workspace_id, 0), COALESCE(namespace_id, 0))
);
```

- 只填 `workspace_id` → 分配给该工作空间（及其下属项目）
- 填 `workspace_id` + `namespace_id` → 分配给该项目

### 端点可见性查询

```sql
SELECT e.* FROM o11y_endpoints e
WHERE e.public = true
   OR e.id IN (SELECT endpoint_id FROM o11y_endpoint_assignments
               WHERE workspace_id = :wsId)
   OR e.id IN (SELECT endpoint_id FROM o11y_endpoint_assignments
               WHERE namespace_id = :nsId)
```

### Host 表扩展

```sql
ALTER TABLE hosts ADD COLUMN
    ssh_port        INT         NOT NULL DEFAULT 22,
    ssh_user        VARCHAR(64) NOT NULL DEFAULT 'root',
    ssh_auth_type   VARCHAR(20) NOT NULL DEFAULT 'password',  -- password / private_key
    ssh_password    VARCHAR(512),
    ssh_private_key TEXT,
    endpoint_id     BIGINT REFERENCES o11y_endpoints(id),     -- 关联的监控端点
    agent_status    VARCHAR(20) NOT NULL DEFAULT 'uninstalled',
    agent_message   VARCHAR(500);
```

- `ssh_password` 和 `ssh_private_key` 存储时加密（AES），API 层返回时脱敏
- `endpoint_id` 直接挂在 host 上
- `agent_status` 追踪 node_exporter + vlagent 的安装状态
- `agent_message` 记录失败原因

### Agent 状态机

```
                       创建主机（未选监控）
                              │
                              ↓
                        uninstalled ←──────── uninstall-agent
                              │
              选了端点 / 点"接入监控"按钮
                              │
                              ↓
                        installing ──失败──→ failed
                              │                 │
                              成功          点"重试"
                              ↓                 │
                         installed ←────────────┘
```

`installed` 表示 node_exporter + vlagent 均安装成功并运行。

## Metrics

### 采集：node_exporter

node_exporter 部署在每台被监控主机上，暴露 `:9100/metrics`。

覆盖指标：CPU、内存、磁盘、网络、文件系统、负载、进程、系统等（几十个 collector、数百个指标）。

### 抓取：VM all-in-one 内置 scrape

VM all-in-one 启动时配置 `-promscrape.config`：

```yaml
# /etc/victoriametrics/scrape.yml
scrape_configs:
  - job_name: "lcp-hosts"
    http_sd_configs:
      - url: "http://lcp-server:8428/api/o11y/v1/discovery/targets"
        refresh_interval: 30s
```

### HTTP SD 接口

LCP 提供 target 列表（`agent_status = installed` 的主机）：

```json
[
  {
    "targets": ["192.168.1.10:9100"],
    "labels": {
      "hostname": "web-01",
      "workspace_id": "1",
      "namespace_id": "2"
    }
  }
]
```

### Metrics 代理查询

前端传 endpointId + PromQL，后端查出 `metrics_url`，转发到 VM：

```
GET /endpoints/{endpointId}/metrics/query_range?query=...&start=...&end=...&step=...
  → 查 o11y_endpoints 获取 metrics_url
  → 转发：GET {metrics_url}/api/v1/query_range?query=...
  → 返回结果给前端
```

VM 集群版需拼租户前缀 `/select/{accountID}/prometheus/api/v1/query`。

### 主机详情页 Metrics Tab

- 时间范围：预设（1h / 6h / 24h / 7d / 30d）+ 自定义日期时间选择器
- 图表库：recharts

#### 指标面板

| 分类 | 指标 | PromQL 示例 |
|---|---|---|
| CPU | 使用率 | `100 - avg(rate(node_cpu_seconds_total{mode="idle",instance="$ip:9100"}[5m])) * 100` |
| CPU | 负载 | `node_load1{instance="$ip:9100"}` / `node_load5` / `node_load15` |
| 内存 | 使用率 | `(1 - node_memory_MemAvailable_bytes/node_memory_MemTotal_bytes) * 100` |
| 内存 | 可用量 | `node_memory_MemAvailable_bytes{instance="$ip:9100"}` |
| 磁盘 | 使用率 | `(1 - node_filesystem_avail_bytes/node_filesystem_size_bytes) * 100` |
| 磁盘 | 读写 IOPS | `rate(node_disk_reads_completed_total[5m])` |
| 磁盘 | 读写吞吐 | `rate(node_disk_read_bytes_total[5m])` |
| 网络 | 收发流量 | `rate(node_network_receive_bytes_total[5m])` |
| 网络 | 收发包量 | `rate(node_network_receive_packets_total[5m])` |
| 系统 | Uptime | `node_time_seconds - node_boot_time_seconds` |
| 系统 | 进程数 | `node_procs_running` / `node_procs_blocked` |

### 主机列表页实时指标

后端批量代理查询，前端 30s 轮询。

```
GET /api/o11y/v1/hosts/metrics?ids=1,2,3&metrics=cpu,mem,disk
```

响应：
```json
{
  "apiVersion": "o11y/v1",
  "kind": "HostMetricsList",
  "items": [
    {
      "hostId": "1",
      "timestamp": "2026-03-13T10:00:00Z",
      "metrics": {
        "cpu_usage": 45.2,
        "mem_usage": 72.1,
        "disk_usage": 55.0,
        "load1": 2.3
      }
    },
    {
      "hostId": "2",
      "metrics": null,
      "error": "endpoint unreachable"
    }
  ]
}
```

后端内部流程：
1. 批量查 hosts 表，拿到 hostname/ip + endpoint_id
2. 按 endpoint_id 分组
3. 并发查询每个 VM 端点（PromQL 用正则批量匹配 `instance=~"host-a|host-b"`）
4. 合并结果按 hostId 组织返回
5. 某个 endpoint 超时/失败 → 对应主机返回 `metrics: null`

前端轮询策略：
- 页面加载 → 立即请求 metrics
- 30s 定时器轮询
- 翻页 → AbortController 取消旧请求，用新页 id 发新请求，重启定时器
- tab 不可见 → `visibilitychange` 暂停轮询
- 请求失败 → 指标列显示 `--`，下次轮询重试

## Logs

### 采集：vlagent

vlagent 部署在每台被监控主机上，采集本机日志文件，推送到 VictoriaLogs。

核心特性：
- 文件 tail（`/var/log/syslog`、`/var/log/messages`、应用日志等）
- 磁盘缓冲（VictoriaLogs 不可用时数据不丢失）
- 支持多目标发送（日志复制）

vlagent 配置文件（LCP 安装时下发到主机）：

```yaml
# /etc/vlagent/config.yml
scrape_configs:
  - job_name: "host-logs"
    file_sd_configs:
      - files:
          - /var/log/syslog
          - /var/log/messages
          - /var/log/*.log
    relabel_configs:
      - target_label: hostname
        replacement: "${HOSTNAME}"
      - target_label: workspace_id
        replacement: "${WORKSPACE_ID}"
      - target_label: namespace_id
        replacement: "${NAMESPACE_ID}"

remote_write:
  - url: "${LOGS_URL}/insert/jsonline"
```

### Logs 代理查询

前端传 endpointId + LogsQL，后端查出 `logs_url`，转发到 VictoriaLogs：

```
GET /endpoints/{endpointId}/logs/query_range
    ?query={hostname="web-01"}
    &start=...&end=...&limit=1000
  → 查 o11y_endpoints 获取 logs_url
  → 转发：GET {logs_url}/select/logsql/query?query=...
  → 返回结果给前端
```

### 主机详情页 Logs Tab

- 位置：主机详情页新增 Logs tab（与 Metrics tab 并列）
- 日志列表：时间戳 + 日志内容，倒序排列
- 搜索：LogsQL 语法，支持关键字搜索、字段过滤
- 时间范围：与 Metrics tab 共享预设时间选择器
- 实时：可选自动刷新（10s / 30s / 关闭）

### 安装流程

借助 `lib/ansible/` 已有的 Ansible 执行引擎（PlaybookExecutor + SSHConnector + command/template module），安装过程由内嵌 Playbook 驱动。

```
POST /hosts/{id}/install-agent { endpointId: 5 }
  │
  ├── 1. 校验：查 host、endpoint 是否存在
  ├── 2. 更新 DB：agent_status = installing, endpoint_id = 5
  ├── 3. 返回 202 Accepted
  │
  └── 4. 异步 goroutine:
        │
        ├── 从 DB 读取 host 的 SSH 凭证 + endpoint 的 URL
        │
        ├── 构建 Ansible Inventory:
        │     hosts:
        │       "192.168.1.10":
        │         port: 22
        │         remote_user: root
        │         password: "xxx"           # 或 private_key_content
        │         hostname: "web-01"
        │         repo_url: "http://..."    # 包仓库地址（来自 LCP 配置）
        │         os_family: "RedHat"       # 或 "Debian"
        │         metrics_url: "http://vm:8428"
        │         logs_url: "http://vlogs:9428"
        │
        ├── 加载内嵌 Playbook（go:embed）
        │     source := project.NewBuiltinSource(playbookFS, "playbooks")
        │
        ├── PlaybookExecutor.Execute()
        │     ├── 自动建 SSH 连接（SSHConnector，基于 Inventory 中的凭证）
        │     ├── 按 Playbook 中的 task 顺序执行
        │     └── 返回 PlaybookResult{Success, Stats, Error}
        │
        └── 根据 PlaybookResult 更新 DB:
              成功 → agent_status = installed, agent_message = ""
              失败 → agent_status = failed, agent_message = result.Error
```

#### install_agent.yml

```yaml
- name: Install monitoring agent
  hosts: all
  become: true
  tasks:
    # ============ 配置包源 ============
    - name: Configure LCP YUM repo
      copy:
        content: |
          [lcp-packages]
          name=LCP Packages
          baseurl={{ repo_url }}
          enabled=1
          gpgcheck=0
        dest: /etc/yum.repos.d/lcp.repo
        mode: "0644"
      when: "{{ os_family == 'RedHat' }}"

    - name: Configure LCP APT repo
      command:
        cmd: >
          bash -c 'echo "deb {{ repo_url }} stable main"
          > /etc/apt/sources.list.d/lcp.list && apt-get update'
      when: "{{ os_family == 'Debian' }}"

    # ============ node_exporter ============
    - name: Check if node_exporter is already installed
      command:
        cmd: rpm -q node_exporter
      register: ne_check
      ignore_errors: true

    - name: Install node_exporter (yum)
      command:
        cmd: yum install -y node_exporter
      when: "{{ ne_check.failed and os_family == 'RedHat' }}"

    - name: Install node_exporter (apt)
      command:
        cmd: apt-get install -y node_exporter
      when: "{{ ne_check.failed and os_family == 'Debian' }}"

    - name: Enable and start node_exporter
      command:
        cmd: systemctl enable node_exporter && systemctl start node_exporter

    - name: Verify node_exporter is running
      command:
        cmd: systemctl is-active node_exporter
      register: ne_status
      failed_when: "{{ ne_status.stdout != 'active' }}"

    - name: Verify node_exporter metrics endpoint
      command:
        cmd: curl -sf http://localhost:9100/metrics > /dev/null
      retries: 3
      delay: 2
      until: "{{ not failed }}"

    # ============ vlagent ============
    - name: Check if vlagent is already installed
      command:
        cmd: rpm -q vlagent
      register: vl_check
      ignore_errors: true

    - name: Install vlagent (yum)
      command:
        cmd: yum install -y vlagent
      when: "{{ vl_check.failed and os_family == 'RedHat' }}"

    - name: Install vlagent (apt)
      command:
        cmd: apt-get install -y vlagent
      when: "{{ vl_check.failed and os_family == 'Debian' }}"

    - name: Deploy vlagent config
      template:
        src: templates/vlagent.yml.j2
        dest: /etc/vlagent/config.yml
        mode: "0644"

    - name: Enable and start vlagent
      command:
        cmd: systemctl enable vlagent && systemctl restart vlagent

    - name: Verify vlagent is running
      command:
        cmd: systemctl is-active vlagent
      register: vl_status
      failed_when: "{{ vl_status.stdout != 'active' }}"
```

#### templates/vlagent.yml.j2

```yaml
scrape_configs:
  - job_name: "host-logs"
    file_sd_configs:
      - files:
```

#### Go 代码调用

```go
// pkg/apis/infra/agent_installer.go

//go:embed playbooks
var playbookFS embed.FS

func (inst *AgentInstaller) Install(ctx context.Context, host *DBHost, endpoint *o11y.DBEndpoint) {
    // 1. 构建 Inventory
    inv := ansible.Inventory{
        Hosts: map[string]map[string]any{
            host.IPAddress: {
                "port":                host.SSHPort,
                "remote_user":         host.SSHUser,
                "password":            decrypt(host.SSHPassword),
                "private_key_content": decrypt(host.SSHPrivateKey),
                "hostname":            host.Hostname,
                "repo_url":            inst.config.Packages.RepoURL,
                "os_family":           host.OSFamily,
                "metrics_url":         endpoint.MetricsURL,
                "logs_url":            endpoint.LogsURL,
            },
        },
    }

    // 2. 加载内嵌 playbook
    source := project.NewBuiltinSource(playbookFS, "playbooks")
    data, _ := source.ReadFile("install_agent.yml")
    var playbook ansible.Playbook
    yaml.Unmarshal(data, &playbook.Play)

    // 3. 执行
    var logBuf bytes.Buffer
    exec := executor.NewPlaybookExecutor(inv, source, executor.WithLogOutput(&logBuf))
    result, err := exec.Execute(ctx, &playbook)

    // 4. 更新状态
    if result.Success {
        inst.hostStore.Patch(ctx, host.ID, map[string]any{
            "agent_status":  "installed",
            "agent_message": "",
        })
    } else {
        inst.hostStore.Patch(ctx, host.ID, map[string]any{
            "agent_status":  "failed",
            "agent_message": result.Error,
        })
    }
}
```

### 卸载流程

同样由内嵌 Playbook 驱动：

```yaml
# uninstall_agent.yml
- name: Uninstall monitoring agent
  hosts: all
  become: true
  tasks:
    - name: Stop and remove vlagent
      command:
        cmd: systemctl stop vlagent; yum remove -y vlagent || apt-get remove -y vlagent
      ignore_errors: true

    - name: Stop and remove node_exporter
      command:
        cmd: systemctl stop node_exporter; yum remove -y node_exporter || apt-get remove -y node_exporter
      ignore_errors: true

    - name: Clean up vlagent config
      command:
        cmd: rm -f /etc/vlagent/config.yml
      ignore_errors: true
```

卸载完成后更新 DB：`agent_status = uninstalled, endpoint_id = NULL`。

#### 包仓库配置

LCP 配置文件中指定包仓库地址，Playbook 会自动在目标主机上配置对应的 YUM/APT 源：

```yaml
# config.yaml
packages:
  repoUrl: "http://nexus.internal/repository/lcp-packages"
```

目标主机**无需手动预配置包源**，Playbook 的第一个 task 会自动写入 `/etc/yum.repos.d/lcp.repo` 或 `/etc/apt/sources.list.d/lcp.list`。

#### 离线交付：ISO 方案

适用于客户现场无外网、无 Nexus 的离线环境。

**出厂前制作 ISO**：

```bash
# packaging/iso/build-iso.sh
#!/bin/bash
WORK_DIR=$(mktemp -d)

# 1. 收集所有 RPM
mkdir -p $WORK_DIR/packages/rpm
cp node_exporter-*.rpm vlagent-*.rpm $WORK_DIR/packages/rpm/

# 2. 生成 YUM repo 元数据
createrepo $WORK_DIR/packages/rpm/

# 3. 如需 DEB 支持
mkdir -p $WORK_DIR/packages/deb
cp node_exporter_*.deb vlagent_*.deb $WORK_DIR/packages/deb/
cd $WORK_DIR/packages/deb && dpkg-scanpackages . /dev/null | gzip > Packages.gz

# 4. 打成 ISO
mkisofs -o lcp-packages.iso -R -J $WORK_DIR
```

**客户现场部署**：

```bash
# 1. 挂载 ISO 到任意一台内网机器
mount -o loop lcp-packages.iso /mnt/lcp-packages

# 2. 用 nginx 对外提供 HTTP 访问
# nginx.conf
server {
    listen 8080;
    location /packages/ {
        alias /mnt/lcp-packages/packages/;
        autoindex on;
    }
}

# 3. LCP 配置 repoUrl 指向这台机器
packages:
  repoUrl: "http://repo-server:8080/packages/rpm"
```

后续安装 Playbook 完全不变，`repo_url` 指向客户内网挂载地址即可。

ISO 内容不仅限于 node_exporter 和 vlagent，未来新增的软件（MySQL exporter、Redis exporter 等）都打进同一张 ISO。

**交付物清单**：

```
lcp-packages.iso        # 离线包 ISO
├── packages/
│   ├── rpm/             # RHEL/CentOS 系列
│   │   ├── node_exporter-1.8.0-1.el7.x86_64.rpm
│   │   ├── vlagent-0.x.x-1.el7.x86_64.rpm
│   │   └── repodata/   # createrepo 生成的元数据
│   └── deb/             # Debian/Ubuntu 系列
│       ├── node_exporter_1.8.0_amd64.deb
│       ├── vlagent_0.x.x_amd64.deb
│       └── Packages.gz  # dpkg-scanpackages 生成的索引
```
